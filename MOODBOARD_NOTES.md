# Moodboard Notes

## Observed References

The moodboard combines technical, archival, and high-density visual systems:

- Typewriter-like concrete poetry and text arranged as diagrammatic structure.
- Anatomical and technical manuals with labels, callouts, and scanned-paper texture.
- Dense radar/operations software with tables, grids, charts, maps, and highlighted rows.
- Astrological charting with circular systems, colored symbols, and relational lines.
- Circuit-board documentation and schematic diagrams.
- Tangled infrastructure cabling.
- Racing wheel controls with many physical buttons, labels, and color-coded affordances.
- Brain-machine / cybernetic diagrams.
- Seismograph-like stacked signal charts.
- Nike sole technical linework and contour maps.

## Design Ingredients To Carry Forward

- Dense information, but organized into clear zones.
- Labels, metadata, and callouts as part of the visual language.
- Technical precision without making the interface feel cold or generic.
- Strong selected-row treatment inspired by operations consoles.
- Fine-line diagrams, dividers, and coordinate-like structure.
- Small color accents used for meaning, not decoration.
- Tactile controls and command affordances inspired by racing controls.
- Signal/noise visual language for security data and live operations.

## Harshit's Mood Direction

- Complexity should feel intentional, like an expert control surface.
- Thin grids and linework should create clarity.
- Color should be used sparingly and with purpose.
- The interface should reveal system state rather than hide it.
- Interactions should feel immediate and keyboard-first.
- The final walkthrough should directly explain which references shaped which choices.

## What To Avoid

- A generic dark SaaS command palette with only blue glow and blurred glass.
- A literal collage of the references.
- Overloading the palette until it becomes unreadable.
- Decorative diagrams that do not help the user act faster.
- Making the prototype feel like a sci-fi fantasy dashboard instead of a real security tool.

## Possible Visual Direction

Working hypothesis to explore, pending Harshit's direction:

- Neutral base with sharp, utilitarian surfaces.
- Monospace or technical supporting type paired with a readable primary UI font.
- Result rows that behave like instrument-panel selections.
- Thin separators, measurement ticks, status chips, and small metadata.
- Accent colors reserved for severity, selection, and system state.
- Optional paper/manual-inspired details in secondary surfaces or walkthrough framing, not in core usability.
